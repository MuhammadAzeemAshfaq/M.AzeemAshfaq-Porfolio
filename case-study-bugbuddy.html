The Problem

QA processes at most small tech companies were scattered across spreadsheets and Slack threads

Small to mid-size software teams often struggle with QA. Test cases live in Google Sheets, bugs get reported through chat messages, and there is no single place to see the health of a release. This makes it hard for QA leads to report status to stakeholders and easy for critical bugs to slip through unnoticed.

The client needed a dedicated tool built around the software testing lifecycle so their teams could move from chaos to a clear, repeatable process without switching to a heavyweight enterprise platform.

My Approach

Designing around how QA teams actually work

Before writing code I spent time understanding the actual STLC workflow used by the client's QA team. That research shaped the core modules of the platform.

  • A test case manager where QA engineers write, organise, and assign test cases to specific features or sprints, with support for steps, expected results, and attachments.
  • A bug tracker that captures detailed reports with severity levels, environment info, reproduction steps, and assignment to developers, with a clear status pipeline from open to resolved.
  • Sprint integration that links test cases and bugs to active sprints so project managers can see test coverage and bug counts per release in a single dashboard.
  • A reporting layer that generates release readiness reports, showing pass rate, open critical bugs, and test coverage percentage without any manual effort from the QA lead.

One of the most valuable decisions was building a notification system that automatically alerts developers when a bug is assigned to them and pings QA when a fix is marked ready for retest. This alone removed a lot of back-and-forth communication from the team's day.

What I Learned

Key takeaways

Building tools for developers and QA engineers is a different challenge from building consumer products. These users are technical, have high expectations, and will immediately notice inconsistencies in the UI or gaps in the workflow. Close collaboration with the end users throughout development was what made the product actually useful.

Database schema design for this kind of tool requires careful thought about relationships. A bug relates to a test case, a sprint, a developer, and a release all at once. Getting the data model right early saved a lot of refactoring later.
